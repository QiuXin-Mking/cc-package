## 三、关键发现

### 1. 跨模块分析场景效果显著（Q1 +152%, Q4 +95%）

Q1 和 Q4 都是"找一个东西影响了哪些模块"——需要全局视野。无 REPOMAP 时 Claude 逐个模块 grep，反复调工具，有明显的**死亡螺旋**现象：
- Q1 无 REPOMAP 的 Turn 2 单轮耗时 1,582s，是有 REPOMAP 同轮的 3.8 倍
- Q4 无 REPOMAP 的 Turn 2+3 合计 2,610s，是有 REPOMAP 同轮的 2.6 倍

有 REPOMAP 时，Turn 1 读了地图建立全局认知，后续轮次直接复用缓存，Turn 4（总结）仅 33s 和 155s。

### 2. 窄查询和设计类问题反而更慢（Q3 -17%, Q5 -23%）

- **Q3（错误传播）**：是典型的单链路追踪，grep 追踪特定错误码更直接。REPOMAP 是额外开销。
- **Q5（新功能设计）**：本质是推理题，需要的是架构理解而非代码定位。REPOMAP 提供不了帮助。

### 3. 内容质量无明显差异

两组在核心事实（函数名、调用关系、模块列表）的准确性上没有系统差异。差异在于：
- 有 REPOMAP：更偏向结构性描述（"耦合是非对称的"）
- 无 REPOMAP：更偏向具体行号（"ldlm_lock.c:1688"）

### 4. 稳定性差异

有 REPOMAP 组轮间耗时方差更小——Turn 4 都很快（33s-261s），因为前面已经建立了全局认知。无 REPOMAP 组存在 Turn 2/3 的"爆冲"现象。

## 四、适用场景判断

| 场景 | 建议 | 预期效果 |
|------|------|----------|
| 跨模块影响分析（"改 X 会影响哪些文件"） | **强烈推荐** | 省 50-150% 时间 |
| 模块耦合分析（"A 和 B 怎么交互"） | **推荐** | 省 50-100% 时间 |
| 架构全景理解（"这个项目分几层"） | **推荐** | 省 20-30% 时间 |
| 单链路错误追踪（"这个错误码怎么传上来的"） | 不推荐 | 可能更慢 |
| 新功能设计（"加 XX 功能要改哪"） | 不推荐 | 无帮助 |

---

## 五、最佳实践

1. **大项目（>100 文件）先跑一次 `/repomap`**，生成 REPOMAP.md
2. **在 CLAUDE.md 中加入 repomap-rule**，让 Claude 自动判断是否读取
3. **跨模块问题明确要求 Claude 先读 REPOMAP**："先读 REPOMAP.md 了解全局，再分析 XX"
4. **窄查询不要强制用**，让 Claude 自己选 grep 还是 REPOMAP

### 推荐的 CLAUDE.md 规则

```markdown
<!-- repomap-rule -->
## REPOMAP.md

REPOMAP.md at the project root is a structural outline of the codebase.
Read it when the task benefits from a map: broad exploration, "where does X live",
cross-module refactors, onboarding. Skip it for narrow lookups where Grep or a
known file path is faster.
```
